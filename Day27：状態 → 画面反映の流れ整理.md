
Day27 で整理する流れ（超重要）

正しい流れはこれだけです👇

状態（配列）がある
状態を元に画面を作る
状態が変わったら
もう一度画面を作り直す

👉
状態 → 画面
状態 → 画面
を毎回やるだけ。

## 1.基本の書き方（最小構成）

ここでは 型（考え方の型） だけ覚えます。
まだ「アプリを作る」意識は不要です。


**1.状態を用意する**

let todos = ["散歩", "読書"];

todos が 本体データ（状態）
画面はまだ触りません


**2.画面を作る関数を用意する**

function render() {
  console.log(todos);
}

今は console.log でOK です。

重要なのは、

render() は
「状態を見て、画面を作る役」

という役割だけ。


**3.状態が変わったら render を呼ぶ**

todos.push("運動");
render();

流れはこうです👇

todos が変わる
render() を呼ぶ
画面（今回は console）が更新される



**Day27の最重要型（これだけ覚えてください）**

状態を変える
↓
renderを呼ぶ
↓
画面が変わる


DOMでもReactでも
この型は一生同じです。

## 2.練習問題（超ミニ）

やることは考えるだけでOKです。
次のコードがあります。


let todos = [];

function render() {
  console.log(todos);
}

todos.push("勉強");


質問（1つだけ）👇

このままだと
"勉強" は console に表示されますか？
表示される / 表示されない のどちらですか？


解答：表示されない

正解です 💯
表示されません。



**なぜか（超短く）**

todos.push("勉強");

ここでは 状態は変わった けれど、

render();

が 呼ばれていない からです。

👉
状態が変わっても、render を呼ばない限り画面は更新されない



状態を変更しただけでは画面は変わらない。
変更後の状態を画面に反映させるために render() が必要。

Day27：状態 → 画面反映の流れ整理（振り返り）

**今日やったこと・学習したこと**

状態（配列）がアプリの本体で、画面はその結果にすぎないと理解した
状態を変更しても、自動で画面は更新されないことを確認した
変更後の状態を画面に反映するために、必ず render() を呼ぶ必要があると学んだ

正しい流れは
状態変更 → render() 実行 → 画面更新 で固定されると理解した


**重要な気づき（核心）**

画面（DOM）を直接いじるのではなく、
状態だけを変更し、画面は毎回作り直す発想が安全

render() は「状態を完成品として組み立てて表示する役割」であり、
状態変更や判断ロジックは持たない


**疑問に思ったこと・解消したこと**

「なぜ状態変更だけではダメなのか？」
→ 画面は状態の結果であり、再描画しない限り反映されないため

「renderは何をしているのか？」
→ 今の状態をそのまま画面に出しているだけ



ひとことでまとめ

状態を変えたら、画面に反映するために必ず render() を呼ぶ。
画面は状態の結果でしかない。

この理解が、そのまま React の再描画思想につながる。
