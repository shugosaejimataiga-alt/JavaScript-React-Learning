🟦 Day3：状態（State）とは何か（概念だけ）

今日は 画面が“なぜ動くのか”の仕組みを理解する日 です。
コードはまだ書きません。

■ まず超シンプルに説明すると…

State（状態）＝アプリの“本物のデータ”を持っている場所。
UI（画面）は、そのデータを映している “結果” にすぎません。

■ 本質的に大切な考え方

✔ 1) 画面が変わるのは「状態」が変わるから

数字・入力値・進捗率など、
アプリの中で “変わる可能性があるもの” はすべて State で管理します。

UI は本物ではなく、
State の中身を見て表示しているコピー です。



✔ 2) UI を直接いじるとアプリは壊れやすくなる

もし画面を手作業で書き換えると…

書き換え忘れ
別の画面が古い情報のまま
更新順がズレる

などのバグが起きやすくなります。

複雑なアプリほど破綻します。



✔ 3) State に触れば、React が UI を全部直してくれる

フロントエンドは本来、

「変わるデータ」を1か所で管理し、
React に画面更新を任せる設計

を目指します。

自分で DOM を書き換えなくても、
State が変わった瞬間に React が UI を再描画して整えてくれる 仕組みです。



■ 今日の概念まとめ

“画面が動く理由” は、State が変わるから

UI を直接触らず、「本物のデータ（State）」を変えるだけで良い

状態管理の発想は React を理解する前提になる

✅ 結論

State の中に HTML や CSS は一切書きません。
State は “値” だけ を持ちます。


🟦 State に入るもの

例えば：

数字（0, 1, 2…）
文字列（"こんにちは"）
オブジェクト（{ title: "夢", progress: 40 }）
配列（[1, 2, 3]）

つまり 「データ」だけ。



🟥 State に入らないもの

HTML
CSS
ボタンの構造
デザイン
見た目の配置

これらは UI（画面）側に書きます。

State はあくまで

本物のデータを持つ倉庫

であり、
UI はその倉庫の中身を読んで表示するだけ。

State は「変わる値だけ」を置く場所で、HTML は書かない。
