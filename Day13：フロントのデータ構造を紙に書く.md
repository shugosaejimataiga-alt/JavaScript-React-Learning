🟦 Day13：フロントのデータ構造を紙に書く（概念だけ）

今日は まだコードを書かない日 です。
フロントエンドが扱う「データの本体」を、画面から切り離して整理しました。

■ まず超シンプルに説明すると…

フロントエンドでは、
画面より先に「データの形」を決める必要がある
ということを学びました。



■ 本質的に大切な考え方

✔ 1) データが本体で、画面は結果

画面に見える
進捗バー・色・アイコンなどは本体ではなく、
データから計算・変換された結果である。



✔ 2) Dream と Action は 1対多の関係

1つの Dream に対して

複数の Action が紐づく

Dream は「目的」、
Action は「現実の行動」という役割分担になっている。



✔ 3) 進捗は Dream が直接持つものではない

Dream の進捗は保存しない。
Action がどれだけ実行・完了されたか
その集合から計算される。

つまり、

Dream = 意味・理由・期限などの入力データ

Action = 実行状況（進捗）を持つデータ

進捗バー = Action の状態から導かれる結果

という構造になる。



■ 今日理解したことの核心

フロントエンド設計では、
「何を保存するか」と
「何を計算して表示するか」を
明確に分けて考える必要がある。

これは後の React の state 設計に直結する考え方。

ここまでが Day13の概念整理 です。
