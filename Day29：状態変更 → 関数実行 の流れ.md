
## 1.今回の学習の説明（まず考え方だけ）

結論だけ先に言います。

状態（データ）を変えただけでは、画面は変わらない
**「状態を変えたあとに、必ず何かを呼ぶ」**必要がある
その「何か」が 再描画用の関数（例：render）

つまり今日の主役はこれです👇

状態を変える
↓
関数を呼ぶ
↓
画面が更新される


ここで大事な確認です。
👉 状態変更と画面更新は、同じものではない

## 2.基本の書き方（最小構成だけ）

※「完成コード」ではありません。構造を見るだけです。


let count = 0;   // 状態

function render() {
  console.log(count); // 画面更新の代わり
}

function increment() {
  count = count + 1;  // 状態変更
  render();           // ← ここがDay29の核心
}


ここで見るポイントは 1つだけ。

increment() の中で
状態を変えたあとに、render() を呼んでいる

❌ ダメな考え
状態を変えたら勝手に画面が変わるはず

⭕ 正しい考え
状態を変えたら、更新処理を自分で呼ぶ

まず結論（短く）

✅ 本番（React）でも使う
→「状態が変わったら再描画される」という設計思想

❌ render() を自分で呼ぶコードは書かなくなる
→ Reactが自動でやる

🟦 Day29｜状態変更 → 関数実行 の流れ（振り返り）


■ 今日やったこと・学習したこと

状態（データ）が本体で、DOM（画面）は結果だと再確認した

状態を変更しただけでは画面は変わらず、
状態変更のあとに更新用の関数（render）を呼ぶ必要があると理解した

renderの中ではDOMを一度すべて消して作り直しても問題なく、
それは「DOM操作」ではなく状態からの再生成であると学んだ

イベント内でDOMを直接消すのがダメで、
状態を変更 → 結果として画面が変わるのが正しい設計だと整理できた


■ 書く練習で理解できたこと

「ボタンを押す → 状態が変わる → renderが呼ばれる → 画面が更新される」
という一貫した流れを文章で説明できるようになった

「削除＝DOMを消す」ではなく、
**「状態から対象のデータを除外する」**という考え方が定着した


■ 疑問に思ったこと・解消されたこと

❓「renderの中で全部消しているのに、なぜOKなのか？」

→ DOMを基準にしていないからOK
状態を正として、毎回正しい画面を作り直しているだけ

❓「今日はrenderの中身を学ぶ日なのか？」

→ 違う
今日は「renderをいつ・なぜ呼ぶか」を理解する日だった


■ 今日の核心（1文）

状態を変更したら、必ず更新用の関数を呼び、
画面はその結果として再生成される。
