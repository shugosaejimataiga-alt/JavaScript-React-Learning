🟦 Day8：なぜ React のような仕組みが必要になったか（概念だけ）

今日は まだコードを書かない日 です。
SPA（Single Page Application）が広がる中で、
なぜ React のようなフレームワークが必要になったのかを整理します。

■ まず超シンプルに説明すると…

React は
「UI 更新を全部自動化するための仕組み」
です。

でも本質はもう少し深いです。



■ 本質的に大切な考え方

✔ 1) SPA では UI を全部 JavaScript が管理する必要がある

ページをリロードしないアプリでは、

一覧 → 詳細 → 編集
状態が変わるたび UI を調整
表示とデータの整合性をずっと保つ

これらを 人間が DOM を手で触って管理するのは破綻しやすい。



✔ 2) React は「状態（State）」さえ変えれば UI を自動で作り直す

従来：

要素を追加
並べる
色を変える
数を更新
→ すべて人の手で DOM 操作が必要


React：

状態が変わったら、UI 全部を React が再計算してくれる
→ 更新漏れが起きない
→ コードが圧倒的にシンプルに

つまり React は
UI を人間が管理しなくて良い世界 を作った。



✔ 3) React を学ぶ本当の理由

現代のアプリは、

画面が動きまくる
要素が増えたり消えたり
状態が複雑に変わる
バックエンドと連携する

こうした環境で
“状態が本体で、UI はその結果である”
という設計が必須になる。

それを一番うまく実現するのが React。

ここまでが今日の概念です。



🟦 あなたの理解を整えた Day8 の総まとめ

SPA はページ全体をリロードせず、
必要な部分だけを更新するから早くて便利。

しかしその反面、

画面のあちこちが頻繁に変わる
状態が複雑に分岐する
表示とデータの整合性を常に保たなければならない

そのため、人間が JavaScript で DOM 操作を細かく書き続けるのは非常に大変。
更新漏れやバグが起きやすく、運用・保守も難しくなる。

そこで React のようなフレームワークが必要になった。

React は DOM 操作を人間の代わりに全自動でやってくれる仕組みで、
開発者が考えるべきことは

アプリがどんな状態（State）を持つか
状態がどう変化するか

この2つだけになる。

つまり
「状態の変化を書けば UI は React が勝手に作り直してくれる」
という世界観が React の本質。
