✏️ 練習問題①（基本）
要件

ボタンを1つ作る
そのボタンをクリックしたら
コンソールに clicked と表示する


<button id = button1> クリック </button>

const button1 = document.getElementedById("button1");

button1.addEventListener("click" , function () {
  console.log(クリックしました。);
});



いいですね、書こうとしている方向性は合っています 👍
あとは 3か所だけ修正すれば、正しく動きます。
一気に直さず、1つずつ確認します。

修正①：HTML（idの書き方）

今：
<button id = button1> クリック </button>

👉 idは文字列なので、こう書きます。

<button id="button1">クリック</button>

※ これは HTMLのルールです。



修正②：JavaScript（メソッド名）

今：
document.getElementedById("button1");

👉 正しい名前は getElementById です
（ed は付きません）

document.getElementById("button1");



修正③：console.log の中身

今：
console.log(クリックしました。);

👉 文字は 必ずクォーテーションで囲む

console.log("クリックしました。");

🟦 Day21：イベントとは何か（click）

今日は ユーザー操作と処理をつなぐ仕組みを理解する日です。



■ まず超シンプルに説明すると…

イベント＝「何かが起きたときに処理を動かすためのきっかけ」。

クリック・入力・スクロールなど、
ユーザーの行動そのものがイベントになる。



■ 本質的に大切な考え方

✔ 1) JavaScriptは「自分からは動かない」

JavaScriptの処理は、
イベントが起きるまで実行されない。

クリックされなければ、
書かれている処理は 一切動かない。


✔ 2) イベント登録は「実行」ではなく「予約」

addEventListener は、

「クリックされたら、この処理をやってください」

という 予約をしているだけ。

👉 実行されるのは
実際にクリックが起きた瞬間。


✔ 3) 関数は「その時に動く処理の箱」

clickイベントでは、

外側：イベントの予約

内側（function）：実際に動く処理

という役割分担になっている。

そのため、
処理は 必ず function の中に書く必要がある。


✔ 4) フロントエンド設計につながる考え方

フロントエンドは、

「画面を表示する」だけでなく
ユーザーの行動 → 処理 → 結果
という流れを設計する仕事。

イベントは、その 入口。



■ 今日の理解まとめ（1文）

イベントとは、ユーザーの行動をきっかけに処理を実行するための「予約の仕組み」。
